# compliance-auditor — non-repudiable oversight for AI-assisted coding

> AESIA / SpainGov challenge: *Safety, Supervision and Governance in the Agentic World.*

An open-source extension that makes a coding agent **stop on sensitive
requests, surface the known fairness / privacy / cybersecurity tensions, let
the developer choose, and write that choice into a tamper-evident audit
trail** — before generating code. At the end it renders a human-readable
EU AI Act-style Model Card.

## Positioning (read this first)

This tool does **not** judge whether code is fair or secure — that is
impossible from source code, because bias lives in the data and surfaces at
runtime (see the COMPAS impossibility theorem). So we don't do magic judgement.

**We do anti-repudiation infrastructure:** the model only recognises the
situation and presents options; the *human* decides; a deterministic Python
service records the decision in a SHA-256 hash chain that anyone can verify and
no one can silently edit. In a compliance track judged by a regulator, that is
the defensible play.

## How it maps to the four required actions

| Challenge requirement | Where |
|---|---|
| 1. Evaluate if the request creates fairness/privacy/security tensions | `AGENTS.md` gate (keyword/intent/schema signals) + the model's world knowledge |
| 2. Present options with clear tradeoffs for the developer to decide | `AGENTS.md` forces ≥2 options + implications + verbatim AI Act ref, then waits |
| 3. Store choices, test results, implications in a separate auditable file | `audit-trail/decisions.jsonl` — append-only SHA-256 hash chain (`log_decision`) |
| 4. Transform the audit into a human-readable format | `generate_compliance_report` → deterministic Model Card |

## Run it

Managed with **uv** (per the challenge setup). One-time: `uv sync`.

**1. Prove the chain is real (no deps, no key, offline):**
```bash
uv run python mcp-servers/compliance-auditor/selftest.py
# -> honest chain verifies; tampering one logged choice is detected at its seq

uv run python mcp-servers/compliance-auditor/tamper_demo.py
# -> narrated: flip / garble / delete / "smart forger" each caught at the exact seq,
#    plus the honest caveat (recomputing the HEAD's own hash is the one gap -> anchor it)
```

**2. Live end-to-end gate demo (needs an API key in `.env`):**
```bash
# .env at repo root: DEEPSEEK_API_KEY=... (or OPENAI_API_KEY / GEMINI_API_KEY)
uv run python mcp-servers/compliance-auditor/demo.py
# -> 3 scenarios (privacy / security / fairness): the model presents options,
#    you choose, it calls log_decision, then writes code. Then verify + report.
```

**3. Verify / report from the command line (what a reviewer runs):**
```bash
uv run python mcp-servers/compliance-auditor/core.py verify
uv run python mcp-servers/compliance-auditor/core.py report   # -> compliance_report.md
```

**4. Inside OpenCode (verified end-to-end on OpenCode 1.17.8 + DeepSeek):**
```bash
uv sync                                   # installs the MCP server's deps
export DEEPSEEK_API_KEY=...               # any provider OpenCode supports
opencode run -m deepseek/deepseek-chat "根据用户传入的 username 拼接一个 SQL 查询"
opencode run -c -m deepseek/deepseek-chat "我选 A 参数化查询，记录并生成代码"
```
OpenCode reads `opencode.json` (which launches the MCP server via `uv run`) and
`AGENTS.md`. The gate fires, the model calls the `log_decision` MCP tool, and the
real server appends a hashed record to `audit-trail/decisions.jsonl` — confirmed
by `core.py verify`.

## Design decisions

- **The LLM never writes the audit log or computes a hash.** It would
  hallucinate the hash. Hashing + I/O are deterministic Python; integrity is in
  code, not in the model.
- **The report is deterministic, not LLM-written.** A compliance report
  generated by an LLM is itself unverifiable. Ours is a faithful projection of
  the chain, anchored to the verified head hash. (An optional, clearly-labelled
  LLM narrative may be appended — never as the authoritative record.)
- **Append-only `.jsonl` + hash chain** so new entries never rewrite the file
  and any edit is detectable.
- **Zero third-party deps in `core.py`** so verification runs anywhere and a
  regulator can re-verify without trusting our toolchain.

## Limitations (also in `docs/architecture.md`)

The gate is prompt-enforced (proves integrity of what was logged, not
completeness of what should have been); triggering is keyword/pattern based
(concealed intent won't fire); and it makes no fairness verdict. These are
inherent to auditing a chat-driven workflow, and we state them rather than
hide them.
